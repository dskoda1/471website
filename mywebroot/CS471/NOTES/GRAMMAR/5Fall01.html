SGML (Standard Generalized Markup Language)

  • Standard : ISO 8879:1986
  • Generalized : Meta-language (i.e. framework) for constructing markup languages.
  • Markup : Codes added to the stream of an encoded text to signal structure, formatting, or processing commands.
  • Language : Uses Formal Languages to describe markup languages.

XML

  • A subset of SGML which is simpler and more efficient therefore good for processing documents on the web.
  • XML is like HTML because it contains elements and attributes but XML allows the author to change/add elements and attributes without changing the XML standard.  These elements and attributes can be defined internal to the file or external.
  • DTD files or XSL files can be used to define the grammars used to validate XML files.
  • XML Vocabularies-
    • Set of elements and attributes that pertain to a specific application.
    • MathML
    • United States Congress XML and Legislative Documents

DTD -- How to define the structure of an XML document

  •  Document Type Declaration (or Definition), abbreviated DTD provides the grammar for the structure of SGML (or XML) document. 
    • The DTD is essentially a context-free grammar with it own notation for describing variables and productions.
      • i.e. The DTD describes the form of a typical document as well as the tags to be used.
    • Schema's are an alternative technique for providing XML grammars.
  • Example:  HTML 4.01 Strict DTD file at W3
  • HTML is an application of SGML.
  • The grammar of the DTD language is itself a Context Free grammar.  However, we are interested the languages that are  described by the DTD.  Especially how to write a grammar for an XML file.
  • Binding the DTD to the XML document:
    If the DTD is internal to an XML document then form of a DTD is
<!DOCTYPE name-of-DTD [
list of element definitions,  attlist definitions, entity definitions
]>
If the DTD is external then the <!DOCTYPE  statement of the form 
    <!DOCTYPE  name-of-DTD PUBLIC "filename.dtd" "http://URL/filename.dtd">

is in the XML file.  The list of element definitions,  attlist definitions, entity definitions are in the "filename.dtd". This file is located at http://URL/filename.dtd on the web.

Below is the statement declaring the DTD file on local to your system.
   <!DOCTYPE name-of-DTD SYSTEM "filename.dtd">
  • From the DTD's point of view XML files are made up the following simple building blocks.
  • ELEMENT -- defines the "things" your document pertains to, such as customers, orders, students, classes, etc.  Element is used to define the elements in the XML document.
    • Used to define the grammar for parsing (XML/SGML) documents
  • Tags -- used to markup elements.
    • <element_name> marks the beginning of an element
    • </element_name> marks the end of an element
    • Examples:
<ItemCost>  $500  </ItemCost>
<OrderDetails OrderNumber="onid111" OrderDate="1/12/01" />
  • ATTLIST --  Defines attributes and constraints about values that a given element may assume.
  • ENTITY --Used to defined variables that common are common text. It can be used to alias long strings or to include characters in the text that are tokens in grammar.

  • PCDATA -- Parsed character data.  The text found between the start tag and end tag of an XML document.
  • CDATA-- Character data in the attributes of the tag.
  • NOTATION --Used to associate application need to process data i.e. a name avi with a Windows Media Player.

Example: Customer dtd file

  • Note: There is NO space between the "<" and the "!" and the element name.  This is also true for xml files.
  • Here we are only concerned with writting comments in the DTD file and building the grammar for elements in the XML document.
  • Comments begin with --

    • <!-- Customer version 1.0 DTD
    This DTD is an example for the book XML Beginner's Guide, and builds a customers and orders management system.
    Copyright Dave Mercer 2001, 2002k

    -->
  • <! ELEMENT element-name (element description)attribute >
      is the form of DTD "tag"for defining elements in our XML file
    • element-name  is the head of the production rule
    • Element descriptions are essentially regular expressions.
      • are the RHS of the production rules
      • Expressions are composed of
        • other element names
        • special term, #PCDATA, standing for text
      • Operators are:
        • |  denotes "or" (union)
        • , (comma) denotes concatenation
        • Variants of the closure operation for Regular expression
          • *, zero or more occurrences
          • +, one or more occurrences
          • ?, zero or one occurrence
        • Parentheses are used to group operators to their arguments
        • The usual precedence of regular expressions operators are applied
    • There are 4  predefined attributes the DTD ELEMENT can have:
      EMPTY      -- the XML element has no children or text but can have attributes
  • <!ELEMENT OrderDetails EMPTY>
    • CHILDREN -- can have child elements, attributes but NO text
    ANY-- No restriction
    MIXED    -- the XML element can have child elements, attributes AND text
    • Regular Expression specify "possible children node"
      • orderedchildren has exactly two children, first and second.  These children must appear in order specified.
        <!ELEMENT orderedchildren (first,second)>

      • unorderedchildren has exactly one child, two or one.
        <!ELEMENT unorderedchildren (two | one)>

      • manyunorderchildren has zero or more named children others and another in any order.
        < !ELEMENT manyunorderedchildren ( others | another)*>
      • regExp has 4 "named" children, first, maybe, zeroormore, atleastone. These children must appear in order.  There is exactly one first, none or one maybe, zero or more zeroormore , and one or more atleastone.
        <!ELEMENT regExp (first,maybe?,zeroormore*,atleastone+)>

      • Note we can include #PCDATA in any of the above. This is an example of Mixed mode.
  • <!ELEMENT Customer (#PCDATA | Orders)*>
    • "It would be nice if we could specify that an item contains either text, or text followed by one or more list items. But that kind of specification turns out to be hard to achieve in a DTD. For example, you might be tempted to define an item like this: 
    <!ELEMENT item (#PCDATA | (#PCDATA, item+)) >
    That would certainly be accurate, but as soon as the parser sees #PCDATA and the vertical bar, it requires the remaining definition to conform to the mixed-content model. This specification doesn't, so you get can error that says: Illegal mixed content model for 'item'. Found &#x28; ..., where the hex character 28 is the angle bracket the ends the definition." (http://java.sun.com/xml/jaxp/dist/1.1/docs/tutorial/sax/5a_dtd.html)
  • Attributes are information contained in the XML tags
    <!ATTLIST Customer
    •    customerid       ID        #REQUIRED
       phone            CDATA     #IMPLIED
       email            CDATA     #REQUIRED
       fulladdress      CDATA     #IMPLIED>
  • XML Element name 
  • Attribute(s) of this XML element
  • Values/types of these XML attributes 
  • ID -- An unique name across the document
  • CDATA -- Character Data
  • There are more allowable types.

    • Converting the DTD "grammar" to Ordinary CFG 

    • <!ELEMENT OrderItem (ItemCost,  ItemPrice)>   
      is analogous to the production
    OrderItem -->  ItemCost ItemPrice
    • <!ELEMENT OrderItem (ItemCost | ItemPrice)>  
      is analogous to the production
    OrderItem -->  ItemCost  |  ItemPrice
    • The difficult cases involves the use of regular expressions   
    <!ELEMENT OrderItem (ItemCost, Manufacturers+ ,ItemPrice)> 
    is analogous to the productions
    OrderItem -->  ItemCost  Manufacturers ItemPrice  
    Manufacturers --> Manufacturer  | Manufacturer Manufacturers 
    <!ELEMENT OrderItem (ItemCost, Manufacturers*, ItemPrice)>
    is analogous to the productions
    OrderItem -->  ItemCost   Manufacturers  ItemPrice   |  ItemCost    ItemPrice
    Manufacturers     --> Manufacturer  |  Manufacturer Manufacturers

    XML Element and Attribute Name
    • Must begin with a letter, underscore, or a colon followed by any Unicode character that are letters, numbers,hyphens or periods
      • EXCEPT:  can not begin with xml (any combination of cases) 
      • Note: colon used separate namespace and element and attribute name
    Well-formed vs. Valid XML document
    • well-formed 
      • document must contains properly written tags 
      • and properly nested --non-overlapping tag 
      • i.e. tags don't have any special meaning, tags just follows basic XML syntax rules 
    • A valid 
      • document must be not only well-formed but also parsed according to grammar rules in associated DTD (or XML Schema)
    • View XML document in Internet Explore (>6) or Mozilla (>1.3a) to see the Parse tree based on our DTD. This will show if it is well formed.

Together with Edward Mosher and Raymond Lorie he invented the Generalized Markup Language (GML) as a means of allowing the text editing, formatting, and information retrieval subsystems to share documents.
